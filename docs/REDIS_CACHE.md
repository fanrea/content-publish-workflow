# Redis 缓存设计说明

项目里的 Redis 不是主存储，只是读优化层和少量运行时辅助能力。它的设计目标是：

- 减少热点读请求对数据库的压力
- 不让缓存成为主链路单点
- 允许少量短暂不一致
- 缓存出错时不影响主业务

## 一、缓存启用方式

默认情况下：

- `spring.cache.type=simple`

启用 `redis` profile 后：

- `spring.cache.type=redis`

同时 `RedisCacheConfig` 会注册 Redis CacheManager 和错误处理器。

## 二、当前缓存对象

### 1. 草稿详情缓存

用于缓存按主键或业务号查询的草稿详情。

特点：

- TTL 较短
- 不缓存空值
- 适合详情页热点访问

### 2. 草稿列表缓存

用于缓存列表页结果。

特点：

- TTL 很短
- 更适合演示和读多场景
- 因为列表筛选维度多，不建议把它当成强缓存

### 3. 草稿状态统计缓存

用于缓存状态计数。

特点：

- TTL 比详情还短
- 允许轻微过期
- 适合页签计数和统计概览

### 4. 消费去重缓存

用于记录已经消费过的消息标识，避免重复消费。

### 5. 死信扫描标记缓存

用于辅助死信扫描和恢复逻辑。

## 三、TTL 配置

当前支持的配置项：

- `workflow.cache.keyPrefix`
- `workflow.cache.draftDetailTtl`
- `workflow.cache.draftListTtl`
- `workflow.cache.draftStatusCountTtl`

默认值大致是：

- 详情缓存：30 秒
- 列表缓存：5 秒
- 状态统计缓存：10 秒

这个策略明显偏保守，原因是工作流状态变化比较频繁，文档刻意避免让缓存与真实状态脱节太久。

## 四、键前缀设计

项目使用统一前缀隔离环境和服务：

- 默认前缀：`cpw:`

最终键格式类似：

```text
cpw:cache:缓存名:业务键
```

这样做的价值是：

- 多环境共享 Redis 时不易冲突
- 不同服务之间可以隔离
- 运维查键时更容易识别来源

## 五、序列化策略

Redis 值使用 JSON 序列化，主要特点是：

- 可读性比 JDK 序列化好
- 支持时间类型
- 支持非最终类对象的类型信息

这让缓存内容更容易排查，也更适合演示。

## 六、为什么缓存异常不直接抛出

项目为缓存配置了 `CacheErrorHandler`。发生以下异常时：

- `GET`
- `PUT`
- `EVICT`
- `CLEAR`

处理方式是：

- 打日志
- 不阻断主业务流程

原因很直接：

- 缓存是优化层，不是事实来源
- 数据库才是最终真相

如果缓存一出问题就让发布、审核、查询全部失败，架构上就本末倒置了。

## 七、什么时候适合开 Redis

适合：

- 列表和详情读取频率高
- 要演示缓存和就绪探针
- 需要消费去重缓存

不适合：

- 只做最小本地调试
- 不想引入额外依赖

## 八、使用建议

- 读多场景优先看是否真的命中详情和统计缓存
- 不要把业务一致性建立在缓存命中上
- 列表缓存 TTL 不要设太长
- 状态统计缓存要接受短暂延迟
