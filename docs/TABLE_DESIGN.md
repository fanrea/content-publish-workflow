# Table Design

This document explains the MySQL schema in `sql/schema.sql` and the key invariants behind it.
The schema is designed for a workflow system, not just CRUD.

中文说明（简版）：
- 本服务以 MySQL 为主存储（source of truth），Redis 仅作为可选缓存层。
- `content_draft` 是“可变工作副本 + 当前工作流状态”；`content_publish_snapshot` 是“不可变已发布快照”。
- 发布/回滚都不会覆盖历史版本，只会生成新的 `published_version`，便于审计与追溯。
- 表结构的重点在“工作流约束 + 可追踪性 + 可补偿任务”，而不是把所有字段堆进一张表。

## `content_draft`
Mutable working copy and the workflow "current state".

Core columns:
- `biz_no`: business identifier (unique), stable across versions
- `draft_version`: incremented on each edit; used for audit and optimistic concurrency (later)
- `published_version`: last published version number (monotonic)
- `workflow_status`: current state (`DRAFT`, `REVIEWING`, `APPROVED`, `PUBLISHED`, ...)
- `current_snapshot_id`: latest published snapshot id (nullable before first publish)
- `last_review_comment`: review rejection reason for editor feedback (nullable)

Indexes:
- `uk_content_draft_biz_no (biz_no)`
- `idx_content_draft_status (workflow_status)`
- `idx_content_draft_updated_at (updated_at)` / `idx_content_draft_created_at (created_at)`
- `idx_content_draft_status_updated (workflow_status, updated_at)`
- `idx_content_draft_status_created (workflow_status, created_at)`

说明：
- 列表页最常见的访问模式是“按状态筛选 + 按更新时间排序分页”，因此提供组合索引。
- `keyword` 的模糊匹配（`LIKE %kw%`）通常无法有效使用普通索引；如需高质量检索，建议接入 ES 或 MySQL FULLTEXT（需额外评估中文分词）。

## `content_review_record`
Append-only review history.

Core columns:
- `draft_id`, `draft_version`: ties the review to a specific draft content version
- `decision`: `APPROVE` / `REJECT`
- `comment`: optional
- `reviewed_at`

Indexes:
- `idx_review_draft_version (draft_id, draft_version)`
- `idx_review_draft_time (draft_id, reviewed_at)`

## `content_publish_snapshot`
Immutable released content for each published version.

Core columns:
- `published_version`: version number shown to users/operators
- `source_draft_version`: which draft version produced this snapshot
- `rollback_flag`: whether this snapshot is generated by rollback

Constraints / indexes:
- `uk_snapshot_draft_published_version (draft_id, published_version)` ensures one snapshot per version
- `idx_snapshot_draft_time (draft_id, published_at)`

## `content_publish_task`
Side-effect orchestration records (e.g. search refresh, downstream notifications).
This is the foundation for async execution + retries later.

Core columns:
- `task_type`: task category
- `task_status`: `PENDING` / `RUNNING` / `SUCCESS` / `FAILED` (status enum lives in code)
- `retry_times`, `error_message`

Constraints / indexes:
- `uk_task_draft_version_type (draft_id, published_version, task_type)` supports idempotent task creation
- `idx_task_status (task_status)` and `idx_task_draft_status (draft_id, task_status)` for polling/retry

## `content_publish_log`
High-level operator audit log (submit/review/publish/rollback).

Indexes:
- `idx_log_draft_time (draft_id, created_at)`

## Design Highlights
- Snapshots are immutable. Fixes and rollbacks create a new published version.
- Review records are append-only and can be used for audits.
- Tasks are separated from the core draft record to support retries and async workers.

## Notes
- `sql/schema.sql` is a dev reset script (drop and re-create). It is not a migration system.
- Spring Boot embedded DB 初始化使用 `src/main/resources/sql/schema.sql`（语法更偏兼容，便于 H2）。MySQL 以 `sql/schema.sql` 为准。
